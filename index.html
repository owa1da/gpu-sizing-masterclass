Module 01 · Foundations

An AI model is just a giant pile of numbers

Before any equation, you need three ideas. Get these and the rest is arithmetic.

IDEA 1

Parameters = weights

A model is a long list of numbers called parameters (a.k.a. weights). “A 1B model” means 1 billion numbers. To write each new word, the GPU must read those numbers. That single fact drives almost everything.

IDEA 2

We're sizing for serving

This class is about inference — running a finished model to answer users. Training a model needs far more memory (optimizer states, gradients) and is a different calculation. Everything here = inference/serving.

IDEA 3

Two specs rule them all

Not “cores.” For LLMs the two specs that decide everything are memory capacity (does it fit?) and memory bandwidth (how fast can the chip re-read the model every token?). Compute (TFLOPS) matters mostly for prompts.

The mental picture

Think of the model as a huge reference book. To produce every single word, the GPU flips through the entire book. A bigger book (more parameters) = slower per word. A faster reader (more bandwidth) = quicker. The desk it reads on (memory) must be big enough to hold the book and its notes.

Where GPU cores fit

GPU cores (NVIDIA's CUDA & Tensor Cores, Apple's GPU cores) are the tiny units that do the math; their useful, comparable measure is compute (TFLOPS) — which shows up as peak_FLOPS in Equations 3–4, driving prefill/TTFT and big-batch throughput. But there's no rule like “5 cores per 1B”: chat decode waits on memory (bandwidth-bound), so cores often sit idle, and counts aren't comparable across chips. Size by VRAM, bandwidth, and measured TTFT / tok-s — what the next equations give you.

Module 02 · Hardware

VRAM, RAM, unified memory & disk — what each really does

You said you weren't sure how these differ. Here's the whole story in one diagram.

The one rule that matters

The model's weights + every active user's context must fit inside GPU memory. If it spills into system RAM or disk (“offloading”), generation slows to a crawl. Fit in VRAM = fast. Spill = broken.

Desk · shelf · warehouse

VRAM is your desk — tiny but instant; the GPU only works from here. System RAM is the shelf behind you. Disk is the warehouse across town. Disk size never speeds up generation — it just has to hold the files.

NVIDIA & most GPUs

Separate VRAM

Discrete GPUs have their own dedicated memory (VRAM), physically separate from CPU RAM and much faster. The model must fit in VRAM. CPU RAM only helps load files, tokenize, queue requests and run the server.

Apple Silicon

Unified memory

Apple chips share one pool between CPU and GPU — no separate VRAM. So a 128 GB Mac can dedicate most of it to a model. Trade-off: lower bandwidth than top NVIDIA cards, and you must subtract what macOS uses.

Equation 01 · Memory

Equation 1 — Model weights

The first and most reliable number. It's not a guess — it's multiplication.

Every parameter is stored as a number that takes a fixed amount of space, set by its precision. FP16 uses 2 bytes. So the memory just for the weights is:

weights_GB = params_B × bytes_per_weight

That's the “real base” you wanted: 1B × 2 bytes = 2 GB. Quantization simply shrinks bytes_per_weight — trading a little quality for a lot of memory.HF

Accuracy: ~95%

This is the most trustworthy number in the whole class. If it says 16 GB, reality is ~16 GB.

Equation 02 · Memory

Equation 2 — The KV cache (the hidden cost)

The number beginners forget — and the reason your “16 GB model” suddenly needs 50 GB.

As the model generates, it remembers every previous token's “keys and values” so it doesn't recompute them.HF This KV cache lives in GPU memory and grows with users × context length:

# bytes per token, per user kv/token = 2 × layers × kv_heads × head_dim × kv_bytes # the leading 2 = one Key + one Value

Then total KV = users × context_tokens × kv/token. Modern models use GQA (few kv_heads) which slashes this — and KV can be quantized to FP8 to halve it again.

Why it explodes

Weights are shared by everyone. KV cache is per user, per token. Double the users or double the context → double this cost.

Worked: Llama 3.1 8B

From its config: 32 layers, 8 KV heads, head-dim 128, FP16.cfg

kv/token = 2×32×8×128×2 = 131,072 bytes ≈ 0.131 MB / token 50 users × 4,096 tokens = 26.8 GB of KV cache

So your 8B isn't 16 GB

16 GB weights + 27 GB KV + overhead ≈ ~45 GB. An 80 GB GPU fits it easily; a 24 GB card never could.

Equation 03 · Speed

Equation 3 — Throughput (tokens / sec)

Now the speed half. The surprise: for one user, speed is set by bandwidth, not raw compute.

To make each token, the GPU reads all the weights from memory. So a single user's speed is a bandwidth division:

tokens/sec (1 user) ≈ MBU × bandwidth / weights_GB

MBU (model-bandwidth utilization) ≈ 0.6 in practice (range 0.5–0.85) — you never get 100% of the spec sheet.NV That's why an H100 (3.35 TB/s) generates far faster than a Mac (≈0.6 TB/s) on the same model.

Batching is the multiplier

When many users run together, the GPU reads the weights once and serves the whole batch — so total throughput climbs almost linearly with users… until compute saturates (the “roofline”). KV-cache memory caps how big the batch can get.

The two regimes

Bandwidth-bound
Small batch — below the crossover B*. Each extra request is almost-free aggregate throughput, the win behind continuous batching.
Compute-bound
Large batch — past B*. The math units are saturated, so total tokens/sec flattens and per-user speed drops.

Accuracy: ~50–70%

This lands the right ballpark and GPU class. For an SLA, benchmark your exact model + engine.

B* = (MFU / MBU) × (peak_FLOPS / bandwidth) × (bytes_per_weight / 2) # peak_FLOPS = the GPU's raw compute speed (spec sheet, e.g. H100 ≈ 989 TFLOP/s) # peak_FLOPS ÷ bandwidth = the FLOP-per-byte "roofline ridge" — where memory-bound flips to compute-bound H100, FP16: (0.4/0.6) × (989 TFLOP/s ÷ 3.35 TB/s) × (2/2) ≈ 197 H100, Q4: (0.4/0.6) × 295 × (0.6/2) ≈ 59

MFU ≈ 0.4 = model-FLOPs utilization (the share of peak compute you actually hit; the compute-side twin of MBU). That's the 0.4 / 0.6 in the numbers above.

Equation 04 · Speed

Equation 4 — Time to First Token

How long until the answer starts. This is the latency users actually feel.

Serving has two phases. Prefill reads the prompt (sets TTFT). Decode streams the answer (Equation 3).vLLM Unlike decode, prefill is compute-bound — it does ~2 FLOPs per parameter per prompt token:

TTFT ≈ queue + (2 × params_B × prompt_tokens) / (MFU × peak_FLOPS) # queue = the wait before the GPU starts your request — ~0 when idle, grows under load

Use MFU ≈ 0.4 for prefill (defined in Equation 3). The takeaway: TTFT scales with prompt length. Short prompt → instant. 16k-token prompt → it's felt.

Levers to cut TTFT

shorter prompts · smaller model · faster GPU compute · less queueing · cache repeated prefixes. Chunked prefill trades a little TTFT for steadier streaming.vLLM

Worked: 8B model on an H100 (≈989 TFLOPS FP16)

# ×1000 bridges billions ↔ TFLOPS 1,000-token prompt: 2×8×1,000 / (0.4 × 989 × 1000) ≈ 0.04 s 16,000-token prompt: 2×8×16,000 / (0.4 × 989 × 1000) ≈ 0.65 s

Reality is higher

Real TTFT adds queueing, scheduling and network. Treat this as the compute floor; production is often 2–5× more under load.

Module 07 · Architecture

Mixture of Experts — “37B with 10B active”

Modern frontier models are often MoE. One rule keeps your math correct.

An MoE model owns many “experts,” but a router sends each token to only a few.HF So “37B total / 10B active” means it stores 37B weights but each token only uses ~10B.

Memory & disk → use TOTAL params_B Speed & TTFT → use ACTIVE params_B

In one sentence

You need memory like a 37B, but each token costs closer to a 10B. MoE buys capacity/quality cheaply on compute — not on memory.

MoE vs dense — same memory, very different speed

The batching catch

With many users at once, different tokens activate different experts — so the GPU ends up touching far more than “active” params. MoE is faster than its total size, but not magically free at scale.

Module 08 · Planning

Context window — allocated vs used

The setting everyone over-buys. Allocated context is a ceiling; used context is the real load.

WHO DECIDES

Three limits

The model's training sets the real ceiling. Your serving engine (e.g. vLLM max_model_len) sets a configured max. The request uses what it actually needs.vLLM

CAN I SET 10M?

Usually no

Attention cost and KV memory grow with length, and models are only trained for a certain range. Beyond it, quality degrades — more VRAM alone won't fix it.

IN THE EQUATION

Plan max, bill actual

Size memory for the worst-case allocated context, but expect real usage to sit near the average actual length.

worst_case_KV = users × max_context actual_KV = users × avg_used_context # Allocated 32k, but average use 4k? # Survive 32k, run near 4k.

Hallucination: difficulty, not percentage

A model at 300k/1M (30% full) is usually riskier than 7k/8k — more distraction, and key facts get “lost in the middle.”arXiv Risk tracks absolute length & retrieval difficulty, not how full the window is.

Module 09 · Hardware

Choosing the box — Apple vs NVIDIA

Same equations, two very different philosophies. Here's the priced lineup and how to think about each.

Apple = simple local capacity

Unified memory still gives you one easy pool for local models, but 2026 memory cuts removed the biggest desktop configs. Lower bandwidth and a Metal/MLX stack mean Apple shines for 1–few users, local/private jobs — not heavy multi-user serving.

NVIDIA = bandwidth & throughput

Far higher bandwidth + the mature CUDA / vLLM / TensorRT-LLM stack make NVIDIA the choice for real multi-user serving, batching and multi-GPU scaling — at much higher cost and power.

Module 10 · Strategy

Pick the smallest model that does the job

Hardware sizing starts with model choice. The discipline: start small, measure, scale only if quality demands it.

Every extra billion parameters costs you memory, money and speed. So the right question isn't “what's the best model?” — it's “what's the smallest model that passes my quality bar?”

The loop

1) Define a quality bar with a small eval set of your real tasks. 2) Try the smallest candidate. 3) Passes? Ship it. Fails? Step up one size. 4) Then quantize and size hardware.

Bigger-at-Q4 beats smaller-at-FP16

Memory-constrained? A 32B at Q4 (≈19 GB) usually beats a 14B at FP16 (≈28 GB) on quality and uses less memory. Quantization is almost always the better trade.
